### 데이터베이스 정리

1. 흐름 
    - DB Model -> Relational Model -> SQL -> 정규화 -> 저장/Index -> Transaction -> 동시성제러 -> 회복

    순으로 이뤄져있다.

데이터란, 현실세계의 값 즉, 사실을 의미하며,
정보는 그러한 데이터를 가공해서 의미있게 만드는 것을 의미한다.
DB는 조직적으로 저장된 데이터의 집합체이며, 그러한 집합체를 관리하는 소프트웨어를 DBMS 라고 부른다.
SQL 은 관계형 DB를 다루는 언어를 뜻한다.
-   여기서 중요한 점은 DBMS 는 데이터 저장/검색/수정/삭제/보안/무결성/동시성제어 그리고 회복을 담당한다.

DataModeling이란,
요구사항을 분석하고 이를 개념적 설계 (ER 다이어그램 으로 작성) 및 논리적 설계(관계형 스키마)를 통해 물리적 설계(저장/Index 등)순으로 진행된다.
- ER 모델의 핵심
    - 객체 : 독립젇으로 존재하는 대상
    - 속성 : 개체의 특성
    - 관계 : 객체간의 연관
    - 카디널리티 : 1:1, 1:N, N:M 관계 수

- 관계형 모델
    - 릴레이션 : 테이블
    - 튜플 : 행
    - 속성 : 열
    - 도메인 : 속성이 가질 수 있는 값의 범위
    - 차수 : 속성의 개수
    - 카디널리티 : 튜플의 개수

예시 )

|학번|이름|학과번호|
|---|---|---|
|2033001|정공주|C01|
|2033002|부라리컷|C02|
|2033003|쌍노무고양이|C01|

이런 표 하나를 DB 에서는 릴레이션(테이블)이라 부른다.
- 릴레이션 = 표 = 데이터를 행과 열로 저장한 구조
- 류플 = 행 = 하나의 데이터 기록
- 속성 = 열 = 데이터의 항목
- 도메인 = 속성(열)이 가질 수 있는 값의 범위 = 예: 학년은 1-4
- 차수 = 열 개수 = 속성의 개수
- 카디널리티 = 행 개수 = 류플의 개수

즉, 
|항목|값|
|---|---|
|릴레이션 이름|학생|
|속성|학번, 이름, 학과번호|
|류플|학생 한 명의 정보|
|차수|3|
|카디널리티|3|

라는 의미가 된다.

키 종류
- 슈퍼키 : 튜플(행)을 유일하게 식별할 수 있는 속성의 결합니다. = 불필요한 속성값도 포함 가능하다.
- 후보키 : 최소성을 만족하는 슈퍼키 = 유일성 + 최소성 으로 이뤄져있다.
- 기본키 : 후보키중 대표로 선택한 키 = Null, 중복은 불가능하다.
- 대체키 : 기본키로 선택되지 않은 키 = 후보에서 탈락
- 외래키 : 다른 릴레이션(표)의 기본키를 참조하는 키 = 테이블간의 관계 표현

SQL 기본 구조
- SELECT : 열 이름
- FROM : 테이블
- WHERE : 조건
- GROUP BY : 그룸기준
- HAVING : 그룸조건
- ORDER BY : 정렬 기준

으로, 실행 순서는 FROM -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY 이다.
- 이 순서가 중요한 이유는 SQL 결과를 예측할 때 이 순서대로 생각해야 하기 때문이다.


    - * WHERE 조건
    = 같다      <> != 같지 않다
    BETWEEN A Anf B A 이상 B 이하       IS NULL NULL 확인
    IN(...) 목록 중 하나        Like 패턴 검색

예를 들어, WHERE name Like('김%') 이라면 김 으로 시작하는 이름
WHERE score BETWEEN 80 AND 90 이라면 80이상 90이하 를 뜻한다.

이를 집계하는 함수도 존재한다.
- COUNT() : 개수
- SUM() : 합계
- AVG() : 평균
- MAX() : 최대값
- MIN() : 최솟값

예 )
- SELECT dept, COUNT(*)
- FROM student
- WHERE grade >= 2
- GROUP BY dept
- HAVING COUNT(*) >= 3
- ORDER BY dept;

이는, 
- FROM student 학생 테이블을 대상으로
- WHERE grade >=2 2학년을 포함한 이상의 학생만을 남긴다.
- GROUP BY debp 그리고 이들을 학과 별로 묶어서
- HAVING COUNT(*) >=3 학과별 학생 수가 3명 이상인 그룹만을 다시 남긴다.
- SELECT dept, COUNT(*) 그 학과와 학생 수를 출력하고
- ORDER BY dept 다시 학과 기준으로 정렬한다.

라는 의미가 된다.

---

#### 중요한 의미로, WHERE 와 HAVING 의 차리를 보자면,
WHERE 은 개별 행 조건이다.
예를 들어, 

- SELECT *
- FROM student
- WHERE grade >= 2;

라고 한다면, 이는 학생 중에서 2학년 이상인 행만 출력한다 라는 의미를 지녔다.
또한 이는 그룹을 만들기 전에 작동한다.

HAVING 은 그룹 조건이다.
예를 들어, 
- SELECT dept, COUNT(*)
- FROM student
- GROUP BY dept
- HAVING COUNT(*) >= 3;

라고 하면, 학과별로 학생 수를 센 뒤, 학생 수가 3명 이상인 학과만 출력한다. /n라는 의미가 되며, HAVING 은 GROUP BY 이루에 작동한다.

        WHERE = 행을 거른다
        HAVING = 그룹을 거른다

JOIN
- 두 테이블을 연결해서 데이터를 조회하는 것을 의미한다.

예를 들어학생 테이블에는 학과 번호만 있고, 학과명은 학과 테이블만 있는 경우,
- 학생 테이블 : 

|학번|이름|학과번호
|---|---|---|
|1|Kim|C01|
|2|Lee|C02|


- 학과 테이블 : 

|학과번호|학과명|
|---|---|
|C01|컴퓨터|
|C02|경영|

학생 이름과 학과명을 같이 보고 싶으면 JOIN이 필요하다.

        SELECT student.name, department.dept_name
        FROM student
        JOIN department
        ON student.dept_id = department.dept_id;

- 결과 : 

|name|dept_name|
|---|---|
|Kim|컴퓨터|
|Lee|경영|


이때, Join에는 종류가 나뉘는데,
1. INNER JOIN - 양쪽 테이블에서 조건이 맞는 행만 출력
2. LEFT OUTER JOIN - 왼쪽 테이블은 모두 출력, 오른쪽은 맞는 것만 출력
3. RIGHT OUTER JOIN - LEFT 와 반대로 적용
4. FULL OUTER JOIN - 양쪽 테이블의 모든 행 출력
5. SEFL JOIN - 같은 테이블끼리 조인

정규화
- 나쁜 테이블을 좋은 테이블로 쪼개는 과정
    - 중복을 줄이고 이상 현상을 막기 위해 테이블을 잘게 나누는 과정

    - 나쁜테이블의 의미 : 데이터가 중복되며 수정할 때 문제가 생긴다.

        |학번|학생이름|과목번호|과목명|교수명|
        |---|---|---|---|---|
        |1|Kim|C101|DB|박교수|
        |1|Kim|C102|Java|이교수
        |2|Lee|C101|DB|박교수|

        - 문제점
            - Kim 반복, DB와 박교수도 반복, 박교수 이름 바뀌면 여러행을 다 고쳐야 하고 이중 하나만 고치게 되면 데이터의 불일치가 발생.

        - 개선방법
            - 학생 테이블 (학번, 학생이름) 으로 그분지어 나누고, 과목테이블(과목번호, 과목명, 교수명) 을 나눈다. 
            수강테이블(학번, 과목번호)로 나눈다.

            이렇게 나누면 중복이 줄어든다.
        

    - 이상현상
        - 삽입이상 : 특정 정보 삽입을 위해 불필요한 정보까지 필요하다.
        - 삭제이상 : 삭제하면 필요한 정보까지 삭제된다.
        - 갱신이상 : 중복된 값 일부만 수정 -> 값의 불일치 발생

함수종속
- 정규화를 이해하려면 함수종속을 알아야 한다.
    - X -> Y     : X값을 알면 Y값이 하나로 결정된다.
    - 그에 대한 예시로, 
    
    학번 -> 학생이름
    
    과목변호 -> 과목명

    학과번호 -> 학과명 

    이 말은 학번을 알면 학생이름이 정해지고 과목번호를 알면 과목명이 정해진다. 

    여기서 우리는 X를 결정자라고 한다.
    따라, 학번 이 곧 결정자가 된다.

---
정규형
- 
|정규형|핵심 내용|
|---|---|
|1NF|원자값 즉, 모든 속성값이 원자값이어야 한다. (원자값 : 더 이상 쪼개지지 않는 하나의 값이다. 따라, 하나에 하나의 데이터만 부여한다.)|
|2NF|부분 함수 종속 제거 (복합키일 때 중요하다.)|
|3NF|이행 함수 종속 제거(중간 단계를 거쳐서 결정되는 것이다.)|
|BCNF|모든 결정자가 후보키여야 한다. (X -> Y 조건 일때, X는 결정자 이자 반드시 후보키어야 한다. 또한, 예를 들어 어떤 속성이 다른 속성을 결정하는데, 그 속성이 후보키가 아니라면 BCNF 위반이다.)|

근데 예를 들어 어떤 속성이 다른 속성을 결정하는데, 그 속성이 후보키가 아니라면 BCNF 위반이다. 라는 얘기는 곧 위반이 가능하다 라는 얘기가 아닐까? 나중에 찾아보자.

트랜잭션
- 데이터베이스의 하나의 작업 단위다.

예를들어, A 계좌에서 10,000원 출금 -> B 계좌에 10,000원 입금 되어야 하는 은행 시스템처럼 이 둘의 작업은 하나의 작업으로 묶여야 한다.

따라서 트랜잭션은 전부 성공하거나 전부 실패애햐 한다.

---
ACID

|성질|의미|예시|
|---|---|---|
|Atomicity원자성|모두 실행 또는 모두 취소|출금/입금 둘다 성공하거나 취소|
|Consistancy 일관성|DB규칙 유지|계좌 잔액 총합이 이상해지면 안됨|
|Isolation 고립성|동시에 실행되어도 간섭없음|두 송금이 서로 꼬이면 안됨|
|Durability 지속성|완료 결과는 보존|COMMIT 후 장애가 나도 결과 유지|

마지막의 Durability 가 의미하는건 COMMIT 후 장애가 나도 결과를 유지해야 한다 = Durability, 지속성 즉, 장애 후 로그를 봤을 때 COMMIT이 있으면 REDO한다. 라는 의미인데, 이는 이따가 이어서 다시 한번 더 정리할때 자세히 적겠다.

동시성 제어
-  동시성 제어는 여러 트랜잭션이 동시에 실행될 때 DB가 꼬이지 않게 하는 기술이다.

예를들어, T1 이 A 계좌를 주정 중이고, T2도 A 계좌를 수령하려고 할때, 둘이 동시에 같은 데이터를 건드리려고 하니 값이 꼬일 수가 있다.
그래서 우리는 Lock 을 걸게되고 이때 종류는 공유락 / 배타락 으로 나뉠 수 있다.

1. 공유락 S-lock : 읽기용 락 - SELECT 에서 사용
2. 배타락 X-lock : 쓰기용 락 - UPDATE, DELETE 에서 사용

다시 말해, 공유학은 같이 읽을 수 있지만, 배타락은 혼자 사용해야 한다.

락 호환성
|요청|기존S-lock|기존X-lock|
|---|---|---|
|S-lock|가능|불가능
|X-lock|불가능|불가능|


2PL : Two Phrase Lcking, 즉 2단계 락킹 규약을 의미한다.
이는 그게 확장 단계, 축소단계로 구분지으며, 락을 얻을 수 있지만 락 반납은 불가능하거나, 락을 반남할 수 있지만 새 락을 못 얻는다는 각가의 유칙이 정해져있다.

이는 직렬가능성을 보장하면서도 동시에 교착상태가 발생할 수 있다.

회복시스템
- 장애가 나더라도 DB를 정상 상태로 되돌리는 기술이다.
앞서 말했던 Undo/Redo 즉, COMMIT 에 대한 내용이다.

DB는 트랜잭션 수행 내용을 로그에 기록하는데, 이때 장애 발생시 로그를 보고 복구를 하게 된다.

- COMMIT
    - 트랜잭션이 정상 완료 됐다는 뜻이며, REDO를 통해 작업을 다시 반영한다는 의미다.

    이렇게 작업하는 이유는
    - COMMIT이 되었다 할 지라도 에러로 인해 결과가 디스크에 완전히 반영되지 않을 수 있다. 그래서 로그를 보고 재반영하는 것이라 볼 수 있다.

    그렇다면 반대로

    - Undo 는 마오ㅓㄴ료 작업을 취소하는 의미다.

    왜 취소를 할까?

    - COMMIT이 없다는 의미는 즉 정상적으로 환료되지 않았다는 뜻이며 동시에 중간 변경된 내용이 DB내 일부에 반영되있을 수 있는 사례를 대비해, 그 변경된 내용을 되돌린다는 의미가 된다.

            <T1, START>
            <T1, A, 100, 70>
            <T1, COMMIT>

            <T2, START>
            <T2, B, 50, 30>

            장애 발생


|트란잭션|사태|처리
|---|---|---|
|T1|COMMIT 있음|REDO|
|T2|COMMIT 없음|UNDO|

    |트리거 Trigger|특정 사건 발생 시 자동 실행되는 객체|

    1. 뷰 View
    - 실제 데이터를 저장하지 않는 가상의 테이블이다.

            학생(학번, 이름, 학과, 전화번호, 주민번호)
    모든 사용자에게 해당 정보 전부를 보여주면 개인정보 보호가 어려워지게 되니 이때 사용된다.

        CREATE VIEW 학생_기본정보 AS
        SELECT 학번, 이름, 학과
        FROM 학생;
        
    이렇게 하면 유저는 학생_기본적보 라는 가상의 테이블을 조회하게 된다.

    이로인한 장점은
    1. 보안성 - 민감한 열을 숨길 수 있다.
    2. 편의성 - 복잡한 질의를 단순하게 사용할 수 있다.
    3. 논리적 독립성 - 원본 테이블 구조 변화릐 영향을 줄일 수 있다.

    하지만 뷰는 실제 테이블이 아닌 가상의 테이블 이므로 모든 뷰에서 삽입/수정/삭제가 자유롭지 못하다.

    2. 저장 프로시져 Stored Procedure
    - 여러 SQL 문장을 하나의 절차로 묶어서 DB에 저장한 것이다.

    쉽게 말하면, 자주 사용하는 SQL 작업 묶음을 DB 안에 저장해두고 호출해서 사용하는 것이라 보면 된다. 예를 들어, 매번 회원 등급을 계산하고 포인트를 수정하는 SQL 을 하나의 절차로 만들어서 사용할 수 있다.

    이의 장점은
    1. 재사용성 - 반복 작업 저장 후 호출
    2. 성능 - DB 내부에서 실행되어 효과적일 수 있다.
    3. 보안 - 직접 테이블 접근하는 대신 프로시저만 허용 가능.
    -



    3. 사용자 정의 함수 Function
    - 입력값을 받아서 결과값을 반환하는 저장 객체다.
    이는 즉, 프로시저와의 차이는 "값을 변환하는가" 가 핵심이 되겠다.

    |구분|프로시져|함수|
    |---|---|---|
    |목적|작업 절차 수행|값 계산 후 반환|
    |변환값|없을 수도 있음|보통 반환값 있음|
    |사용 느낌|명령 실행|계산결과 사용|

    즉 함수는 값을 계산해서 반환하는 저장객체 라는 의미를 지녔다.

    4. 트리거 Trigger
    - 특정 사건이 발생하면 자동으로 실행되는 객체다.
    이는 유니티에서 OnTriggerEnter2D 작업할 때를 생각해보면 이해하기 쉽다.

    트리거는 쉽게 반응하는 사건이 있다.

    |사건|뜻|
    |---|---|
    |INSERT|삽입발생|
    |UPDATE|수정발생
    |DELETE|삭제발생

    이러란 조건들 일때 트리거는 자동으로 실행된다.
    정말로 Unity 게임같다고 느껴지는 부분이 아닐 수 없다.

2. 데이터 저장과 파일 
- DB의 데이터는 결국 디스크 같은 저장장치에 저장된다.
이때, 중요한 단위가 레코드, 블록 그리고 파일이다.
1. 레코드 Record
    - 하나의 튜플이 실제 저장장치에 저장된 형태라고 보면 된다.
    - 관계형 모델 : 튜플 = 행
    - 저장 관점 : 레코드 = 저장장치에 저장된 한 건의 데이터

예를 들어

    |학번|이름|학과|
    |---|---|---|
    |202234|Kim|컴퓨터|
이 한 행이 저장장치에서는 하나의 레코드로 저장될 수 있다.

2. 블록 Block
- 디스크와 메모리 사이에서 데이터를 주고받는 기본 단위다.
#### 중요한 이유
- DB는 데이터를 한 행씩 가져오는데 아니라, 보통 블록 단위로 읽고 쓴다. 즉, 원하는 데코드 하나를 찾더라도, 실제로는 그 레코드가 포함된 블록을 읽는다고 볼 수 있다.

이에 따른 핵심은 블록 = 디스트 입출력의 기본 단위 라 할 수 있다.

3. 파일 File
- 여러 레코드들의 집합니다.

이 레코드들이 모여서 파일을 구성하고 그 파일은 디스크에 저장된다.

위의 세가지를 정리하자면

|개념|의미|
|---|---|
|레코드|한 건의 데이터|
|블록|디스크 입출력 단위|
|파일|레코드들의 집합|

3. 인덱싱 Indexing
- 검색 속도를 높이기 위한 자료구조다.

쉽게 말하자면 책의 목차라 생각하면 된다.
책 전체를 처음부터 끝까지 보면서 원하는 페이지를 찾기란 쉽지 않지만, 목차를 보면 원하는 페이지를 빠르게 확인 및 찾을 수 있다.

이는 DB의 인덱스도 마찬다지다.

그러다면 그에 따른 장점과 단점이 있을 것이다.

1. 장점으로는 검색 속도의 향상
2. 단점으로는 삽입/수정/삭제 시 인덱스도 함께 갱신해줘야 한다.

즉, 인덱스가 많다고 무조건 좋은 건 아니라 볼 수 있다.

데이터 조회는 빨라질 수 있지만, 데이터 변경은 느려질 수 있다.

인덱스에는 실행되는 순서가 존재한다.
- 순서 인덱스 라고 한다.

이 순서 인덱스는 키 값이 정력된 상태를 기반으로 만들어진 인덱스이다.

예를 들어서 학번 순서대로 정렬된 인텍스
- 정력되어 있기에 범위 검색에 유리하다.

        WHERE 학번 BETWEEN 202234 AND 20203602
    이런 검색에 탁월하다 볼 수 있다.

- 밀집 인덱스와 희소 인덱스
    - 
    - 밀집 인덱스 Dense Index
        - 모든 레코드에 대한 인덱스 항목이 존재한다.
    레코드 하나에 인덱스 하나가 부여되는데, 역시나 이 또한 검색이 빠르다는 장점이 있지만, 단점으로는 인덱스 크기가 커져 갱신 부담이 증가된다.

    - 희소 인덱스 Sparse Index
        - 일부 레코드에 대해서만 인덱스 항목이 존재한다.
        이는 대표 레코드에만 인덱스를 부여하는데, 장점으로는 인덱스의 크기는 작지만, 단점으로는 검색 시 추가 탐색이 필요하다.


이를 표로 비교하자면, 

|구분|밀집 인덱스|희소 인덱스|
|---|---|---|
|인덱스 항목|모든 레코드|일부 레코드|
|크기|크다|작다|
|검색|빠르다|추가 탐색 필요|
|갱신부담|크다|상대적으로 작다|


- B+트리 인덱스
    - DB 인덱스에서 자주 쓰이는 균형 트리 구조다.

    이는 쉽게 균형 기반 인덱스, 범위 검색에 유리 하며 그 특징으로는 

    1. 균형 트리 - 루트에서 리프까지 길이가 일정하다.
    2. 뛰어난 검색 효율성 - 다량의 데이터에서도 빠른 검색 가능.
    3. 범위 검색 유리 - 리프 노드가 연결되어 순차 접근 가능.
    4. DB 인덱스에서 자주 사용 - 실무 DB 인덱스의 대표구조.

    라 볼 수 있다.

        WHERE age BETWEEN 20 AND 30

    이런 식의 범위 검색에 탁월하다.

해싱 Hashing
---
- 해시 함수를 이용해 키 값을 저장 위치로 변환하는 방법이다.
이는 인덱스가 목차라면, 해싱은 사물함 번호를 바로 계산하는 느낌이라 볼 수 있다.

예를 들어, 
        
    해시함수(학번) -> 버킷 번호
학번을 넣으면 저장 위치가 바로 계산된다.

그러면 역시 해시도 함수가 존재할 것이다.

우리는 이를 해시 함수라 부르기로 했다.

1. 해시 함수
    - 키 값을 입력받아 버킷 주소를 계산한다.

            h(key) = bucket address
            이 기본 값으로 예시를 들자면

            h(202223) = 3번 버킷
    그렇다면 버킷이란 무엇일까?

2. 버킷
- 레코드가 저장되는 공간이다.

    버킷 = 해시 결과에 따라 레코드가 저장되는 공간이다.

3. 충돌 Collision
- 서로 다른 키가 같은 버킷으로 배정되는 상황이다.

        h(2023001) = 3
        h(2023011) = 3
이런 식으로 서로 다른 학번임에도 불구하고 동일한 버킷으로 출력되는 걸 의미한다.

4. 오버플로우 Overflow
- 버킷에 공간이 부족한 상황이다.
    
    - 버킷에 들어갈 수 있는 레코드보다 더 많은 레코드가 몰리면 오버플로우가 발생한다.

그래서 종답적으로 본 해싱의 장단점으로는

|구분|의미|
|---|---|
|장점|동등 검색이 빠름|
|단점| 범위 검색에 약함|
|문제점| 충돌와 오버플로우 처리가 필요하다|

이때의 동등 검색이란
    
    WHERE 학번 = 202234
    해싱에서 유리.

    WHERE 학번 BETWEEN 202234 AND 202602
    해싱에서는 불리.

라고 보면 된다.

트랜잭션 후반 보강
---

이미 앞에서 ACID 와 COMMIT, REDO/UNDO 에 대해서 언급했으니 그 부분은 넘어가고 좀 더 집중적으로 세부적인 내용을 보려고 한다.

1. 직렬 수행과 병행 수행
    - 직렬 수행 - 트랜잭션을 하나씩 순서대로 실행하는 것.
        
            T1 완료 -> T2 실행 -> T3가 겹쳐서 실행

    1. 장점 : 성능 향상, 시스템 자원 활용 증가.
    2. 단점 : 데이터 일관성 깨질 수 있음, 동시성 제어 필요

2. 직렬가능성 Serializability
    - 병행 수행 결과가 어떤 직렬 수행 결과와 같다면 올바른 병행 수행으로 보는 기준이다. 즉, 동시에 실행했지만, 결과가 하나씩 실행한 것과 같으면 OK 이다.

---
동시성 제어 보강
---
- 동시성 제어는 병행 수행 중 데이터 일관성이 깨지는 것을 막는 기술이다.

1. 갱신 손실 Lost Update
    - 두 트란잭션이 같이 데이터를 수정하면서 갱신이 사라지는 문제다.

            A = 100

            T1이 A를 읽음 → 100
            T2가 A를 읽음 → 100

            T1이 A = 90으로 저장
            T2가 A = 80으로 저장
        최종 A는 80이 되고 T1의 갱신 90은 사라지게 된다.
        이런 것을 바로 갱신 손실이다.


2. 모순성 Incomsistency
- 한 트란잭션이 여러 데이터를 읽는 동안 다른 트랜젝션이 일부 데이터를 바꿔서 일관성 없는 값을 읽는 문제다.

        A + B의 합을 계산 중인데
        계산 도중 다른 트랜잭션이 A 또는 B를 변경
    그러면 잘못된 합이 나올 수 있다.

3. 연쇄 복귀 Cascading Rollback
- 한 트란잭션이 아직 COMMIT 되지 않은 다른 트랜잭션의 값을 읽었는데, 그 트랜잭션이 나중에 ROLLBACK되면 같이 되돌려야 하는 문제다.

        미완료 데이터를 믿고 사용했다가 같이 망하는 상황이라고 볼 수 있다...

락 보강
---
- 이전에 정리한 내용과 비슷하지만 조금 더 정리해 보겠다.

1. 공유락 S-lock
    - 읽기용
    - 다른 공유락과 함께 가능
    - 베타락과 함쎄 사용 불가능

2. 배타락 X-lock
    - 쓰기용
    - 다른 어떤 락과도 함께 불가능

호환성

|기존/요청|S-lock 요청|X-lock 요청|
|---|---|---|
|S-lock 보유|가능|불가능|
|X-lock 보유|불가능|불가능|

---

교착상태 Deadlock
---
- 서로가 서로의 자원을 기다리며 멈춘 상태이다.
예를 들면 T1은 A에 락을 걸고 B를 기다리고 T2는 B에 락을 걸고 A를 기다린다.

이 말은 즉, 둘 다 상대가 가진 락을 기다리고 결국에는 아무것도 진행하지 못하게 된다.

따라서 2PL은 직렬가능성을 보장하지만 교착상태가 생길 수 있다.
---

회복 시스템 보강
---
COMMIT 이 있다 없다에 대한 부분은 정리했으니 그 다음에는 Log와 그 이후에 대해 조금 더 정리를 해보려 한다.

1. 로그 Log
- 트랜잭션의 변경 내용을 기록한 것이다. 또한 장애가 발생하면 DBMS는 로그를 보고 복구한다.

        <T1, START>
        <T1, A, old value, new value>
    이 의미는

    |로그|의미|
    |---|---|
    |<T1, START>|T1의 시작|
    |T1, A, 100, 70|A값이 100에서 70으로 변경|
    |<T1, COMMIT>|T1 정상 완료|

2. UNDO 와 REDO 다시 정리

|상황|처리|이유|
|---|---|---|
|COMMIT 있음|REDO|완료된 결과값을 보존해야 한다.|
|COMMIT 없음|UNDO|미완료 작업물은 취소해야 한다.|

---

체크포인트 Checkpoint
---
- 회복 작업량을 줄이기 귀해 특정 시점의 DB 상태를 기록하는 지점이다.

쉽게 말하면 복구할 때 처음부터 다 보지 않도록 중간 저장 지점을 만드는 것이라 보면 된다.

약점 파악
1. SQL 작성 순서 vs 실행 순서
    - 작성 순서: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY  
    실행 순서: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY  

    이를 논리적 처리 순서 라고 한다.

2. 참조 무결성
    1. 기본키는 NULL 불가 - 개체 무결성
    2. 속성 값은 도메인 안에 존재한다 - 도메인 무결성
    3. 외래키는 참조 대상에 존재한다 - 참조 무결성
    4. 모든 결정자는 후보키다 - BCNF
    = 1 - 3번까지는 무결성 제약조건 이다.

3. 정규화
        학생(학번, 이름, 학과번호, 학과명)  
        
        학번 → 이름
        학번 → 학과번호
        학과번호 → 학과명
    이게 이렇게 되면 문제인게,  

        학번 → 학과번호
        학과번호 → 학과명
    이 부분이 문제가 되는데, 이걸

        학번 → 학과번호 → 학과명
    이런식으로 해야 이행 함수가 충족된다.  
    이 말은 즉, 이렇게 하지 않으면 3NF 는 이행되지 않는다 라고 봐야한다.

4. 스케쥴
    1. 직렬 가능 스케줄 - 병행 실행 결과가 어떤 직렬 실행 결과와 같음
    2. 회복 가능 스케쥴 - 읽힌 값을 쓴 트랜젝션이 먼저 COMMIT 된다.
    3. 비연쇄 스케쥴 - COMMIT 된 값만 읽는다.
    4. 엄격한 스케쥴 - COMMIT 전까지 read/write 모두 금지된다.

5. 인덱스/해싱
        
        밀집 인덱스 = 모든 레코드에 인덱스 엔트리
        희소 인덱스 = 일부 레코드에만 인덱스 엔트리
        B+트리 = 범위 검색
        해싱 = 동등 검색, 해시 함수, 버킷
