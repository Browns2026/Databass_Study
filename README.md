### 데이터베이스 정리

1. 흐름 
    - DB Model -> Relational Model -> SQL -> 정규화 -> 저장/Index -> Transaction -> 동시성제러 -> 회복

    순으로 이뤄져있다.

데이터란, 현실세계의 값 즉, 사실을 의미하며,
정보는 그러한 데이터를 가공해서 의미있게 만드는 것을 의미한다.
DB는 조직적으로 저장된 데이터의 집합체이며, 그러한 집합체를 관리하는 소프트웨어를 DBMS 라고 부른다.
SQL 은 관계형 DB를 다루는 언어를 뜻한다.
-   여기서 중요한 점은 DBMS 는 데이터 저장/검색/수정/삭제/보안/무결성/동시성제어 그리고 회복을 담당한다.

DataModeling이란,
요구사항을 분석하고 이를 개념적 설계 (ER 다이어그램 으로 작성) 및 논리적 설계(관계형 스키마)를 통해 물리적 설계(저장/Index 등)순으로 진행된다.
- ER 모델의 핵심
    - 객체 : 독립젇으로 존재하는 대상
    - 속성 : 개체의 특성
    - 관계 : 객체간의 연관
    - 카디널리티 : 1:1, 1:N, N:M 관계 수

- 관계형 모델
    - 릴레이션 : 테이블
    - 튜플 : 행
    - 속성 : 열
    - 도메인 : 속성이 가질 수 있는 값의 범위
    - 차수 : 속성의 개수
    - 카디널리티 : 튜플의 개수

예시 )

|학번|이름|학과번호|
|---|---|---|
|2033001|정공주|C01|
|2033002|부라리컷|C02|
|2033003|쌍노무고양이|C01|

이런 표 하나를 DB 에서는 릴레이션(테이블)이라 부른다.
- 릴레이션 = 표 = 데이터를 행과 열로 저장한 구조
- 류플 = 행 = 하나의 데이터 기록
- 속성 = 열 = 데이터의 항목
- 도메인 = 속성(열)이 가질 수 있는 값의 범위 = 예: 학년은 1-4
- 차수 = 열 개수 = 속성의 개수
- 카디널리티 = 행 개수 = 류플의 개수

즉, 
|항목|값|
|---|---|
|릴레이션 이름|학생|
|속성|학번, 이름, 학과번호|
|류플|학생 한 명의 정보|
|차수|3|
|카디널리티|3|

라는 의미가 된다.

키 종류
- 슈퍼키 : 튜플(행)을 유일하게 식별할 수 있는 속성의 결합니다. = 불필요한 속성값도 포함 가능하다.
- 후보키 : 최소성을 만족하는 슈퍼키 = 유일성 + 최소성 으로 이뤄져있다.
- 기본키 : 후보키중 대표로 선택한 키 = Null, 중복은 불가능하다.
- 대체키 : 기본키로 선택되지 않은 키 = 후보에서 탈락
- 외래키 : 다른 릴레이션(표)의 기본키를 참조하는 키 = 테이블간의 관계 표현

SQL 기본 구조
- SELECT : 열 이름
- FROM : 테이블
- WHERE : 조건
- GROUP BY : 그룸기준
- HAVING : 그룸조건
- ORDER BY : 정렬 기준

으로, 실행 순서는 FROM -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY 이다.
- 이 순서가 중요한 이유는 SQL 결과를 예측할 때 이 순서대로 생각해야 하기 때문이다.


    - * WHERE 조건
    = 같다      <> != 같지 않다
    BETWEEN A Anf B A 이상 B 이하       IS NULL NULL 확인
    IN(...) 목록 중 하나        Like 패턴 검색

예를 들어, WHERE name Like('김%') 이라면 김 으로 시작하는 이름
WHERE score BETWEEN 80 AND 90 이라면 80이상 90이하 를 뜻한다.

이를 집계하는 함수도 존재한다.
- COUNT() : 개수
- SUM() : 합계
- AVG() : 평균
- MAX() : 최대값
- MIN() : 최솟값

예 )
- SELECT dept, COUNT(*)
- FROM student
- WHERE grade >= 2
- GROUP BY dept
- HAVING COUNT(*) >= 3
- ORDER BY dept;

이는, 
- FROM student 학생 테이블을 대상으로
- WHERE grade >=2 2학년을 포함한 이상의 학생만을 남긴다.
- GROUP BY debp 그리고 이들을 학과 별로 묶어서
- HAVING COUNT(*) >=3 학과별 학생 수가 3명 이상인 그룹만을 다시 남긴다.
- SELECT dept, COUNT(*) 그 학과와 학생 수를 출력하고
- ORDER BY dept 다시 학과 기준으로 정렬한다.

라는 의미가 된다.

---

#### 중요한 의미로, WHERE 와 HAVING 의 차리를 보자면,
WHERE 은 개별 행 조건이다.
예를 들어, 

- SELECT *
- FROM student
- WHERE grade >= 2;

라고 한다면, 이는 학생 중에서 2학년 이상인 행만 출력한다 라는 의미를 지녔다.
또한 이는 그룹을 만들기 전에 작동한다.

HAVING 은 그룹 조건이다.
예를 들어, 
- SELECT dept, COUNT(*)
- FROM student
- GROUP BY dept
- HAVING COUNT(*) >= 3;

라고 하면, 학과별로 학생 수를 센 뒤, 학생 수가 3명 이상인 학과만 출력한다. 라는 의미가 되며, HAVING 은 GROUP BY 이루에 작동한다.

        WHERE = 행을 거른다
        HAVING = 그룹을 거른다

JOIN
- 두 테이블을 연결해서 데이터를 조회하는 것을 의미한다.

예를 들어학생 테이블에는 학과 번호만 있고, 학과명은 학과 테이블만 있는 경우,
- 학생 테이블 : 

|학번|이름|학과번호
|---|---|---|
|1|Kim|C01|
|2|Lee|C02|


- 학과 테이블 : 

|학과번호|학과명|
|---|---|
|C01|컴퓨터|
|C02|경영|

학생 이름과 학과명을 같이 보고 싶으면 JOIN이 필요하다.

        SELECT student.name, department.dept_name
        FROM student
        JOIN department
        ON student.dept_id = department.dept_id;

- 결과 : 

|name|dept_name|
|---|---|
|Kim|컴퓨터|
|Lee|경영|


이때, Join에는 종류가 나뉘는데,
1. INNER JOIN - 양쪽 테이블에서 조건이 맞는 행만 출력
2. LEFT OUTER JOIN - 왼쪽 테이블은 모두 출력, 오른쪽은 맞는 것만 출력
3. RIGHT OUTER JOIN - LEFT 와 반대로 적용
4. FULL OUTER JOIN - 양쪽 테이블의 모든 행 출력
5. SEFL JOIN - 같은 테이블끼리 조인

정규화
- 나쁜 테이블을 좋은 테이블로 쪼개는 과정
    - 중복을 줄이고 이상 현상을 막기 위해 테이블을 잘게 나누는 과정

    - 나쁜테이블의 의미 : 데이터가 중복되며 수정할 때 문제가 생긴다.

        |학번|학생이름|과목번호|과목명|교수명|
        |---|---|---|---|---|
        |1|Kim|C101|DB|박교수|
        |1|Kim|C102|Java|이교수
        |2|Lee|C101|DB|박교수|

        - 문제점
            - Kim 반복, DB와 박교수도 반복, 박교수 이름 바뀌면 여러행을 다 고쳐야 하고 이중 하나만 고치게 되면 데이터의 불일치가 발생.

        - 개선방법
            - 학생 테이블 (학번, 학생이름) 으로 그분지어 나누고, 과목테이블(과목번호, 과목명, 교수명) 을 나눈다. 
            수강테이블(학번, 과목번호)로 나눈다.

            이렇게 나누면 중복이 줄어든다.
        

    - 이상현상
        - 삽입이상 : 특정 정보 삽입을 위해 불필요한 정보까지 필요하다.
        - 삭제이상 : 삭제하면 필요한 정보까지 삭제된다.
        - 갱신이상 : 중복된 값 일부만 수정 -> 값의 불일치 발생

함수종속
- 정규화를 이해하려면 함수종속을 알아야 한다.
    - X -> Y     : X값을 알면 Y값이 하나로 결정된다.
    - 그에 대한 예시로, 
    
    학번 -> 학생이름
    
    과목변호 -> 과목명

    학과번호 -> 학과명 

    이 말은 학번을 알면 학생이름이 정해지고 과목번호를 알면 과목명이 정해진다. 

    여기서 우리는 X를 결정자라고 한다.
    따라, 학번 이 곧 결정자가 된다.

---
정규형
- 
|정규형|핵심 내용|
|---|---|
|1NF|원자값 즉, 모든 속성값이 원자값이어야 한다. (원자값 : 더 이상 쪼개지지 않는 하나의 값이다. 따라, 하나에 하나의 데이터만 부여한다.)|
|2NF|부분 함수 종속 제거 (복합키일 때 중요하다.)|
|3NF|이행 함수 종속 제거(중간 단계를 거쳐서 결정되는 것이다.)|
|BCNF|모든 결정자가 후보키여야 한다. (X -> Y 조건 일때, X는 결정자 이자 반드시 후보키어야 한다. 또한, 예를 들어 어떤 속성이 다른 속성을 결정하는데, 그 속성이 후보키가 아니라면 BCNF 위반이다.)|

근데 예를 들어 어떤 속성이 다른 속성을 결정하는데, 그 속성이 후보키가 아니라면 BCNF 위반이다. 라는 얘기는 곧 위반이 가능하다 라는 얘기가 아닐까? 나중에 찾아보자.

트랜잭션
- 데이터베이스의 하나의 작업 단위다.

예를들어, A 계좌에서 10,000원 출금 -> B 계좌에 10,000원 입금 되어야 하는 은행 시스템처럼 이 둘의 작업은 하나의 작업으로 묶여야 한다.

따라서 트랜잭션은 전부 성공하거나 전부 실패애햐 한다.

---
ACID

|성질|의미|예시|
|---|---|---|
|Atomicity원자성|모두 실행 또는 모두 취소|출금/입금 둘다 성공하거나 취소|
|Consistancy 일관성|DB규칙 유지|계좌 잔액 총합이 이상해지면 안됨|
|Isolation 고립성|동시에 실행되어도 간섭없음|두 송금이 서로 꼬이면 안됨|
|Durability 지속성|완료 결과는 보존|COMMIT 후 장애가 나도 결과 유지|

마지막의 Durability 가 의미하는건 COMMIT 후 장애가 나도 결과를 유지해야 한다 = Durability, 지속성 즉, 장애 후 로그를 봤을 때 COMMIT이 있으면 REDO한다. 라는 의미인데, 이는 이따가 이어서 다시 한번 더 정리할때 자세히 적겠다.

동시성 제어
-  동시성 제어는 여러 트랜잭션이 동시에 실행될 때 DB가 꼬이지 않게 하는 기술이다.

예를들어, T1 이 A 계좌를 주정 중이고, T2도 A 계좌를 수령하려고 할때, 둘이 동시에 같은 데이터를 건드리려고 하니 값이 꼬일 수가 있다.
그래서 우리는 Lock 을 걸게되고 이때 종류는 공유락 / 배타락 으로 나뉠 수 있다.

1. 공유락 S-lock : 읽기용 락 - SELECT 에서 사용
2. 배타락 X-lock : 쓰기용 락 - UPDATE, DELETE 에서 사용

다시 말해, 공유학은 같이 읽을 수 있지만, 배타락은 혼자 사용해야 한다.

락 호환성
|요청|기존S-lock|기존X-lock|
|---|---|---|
|S-lock|가능|불가능
|X-lock|불가능|불가능|


2PL : Two Phrase Lcking, 즉 2단계 락킹 규약을 의미한다.
이는 그게 확장 단계, 축소단계로 구분지으며, 락을 얻을 수 있지만 락 반납은 불가능하거나, 락을 반남할 수 있지만 새 락을 못 얻는다는 각가의 유칙이 정해져있다.

이는 직렬가능성을 보장하면서도 동시에 교착상태가 발생할 수 있다.

회복시스템
- 장애가 나더라도 DB를 정상 상태로 되돌리는 기술이다.
앞서 말했던 Undo/Redo 즉, COMMIT 에 대한 내용이다.

DB는 트랜잭션 수행 내용을 로그에 기록하는데, 이때 장애 발생시 로그를 보고 복구를 하게 된다.

- COMMIT
    - 트랜잭션이 정상 완료 됐다는 뜻이며, REDO를 통해 작업을 다시 반영한다는 의미다.

    이렇게 작업하는 이유는
    - COMMIT이 되었다 할 지라도 에러로 인해 결과가 디스크에 완전히 반영되지 않을 수 있다. 그래서 로그를 보고 재반영하는 것이라 볼 수 있다.

    그렇다면 반대로

    - Undo 는 마오ㅓㄴ료 작업을 취소하는 의미다.

    왜 취소를 할까?

    - COMMIT이 없다는 의미는 즉 정상적으로 환료되지 않았다는 뜻이며 동시에 중간 변경된 내용이 DB내 일부에 반영되있을 수 있는 사례를 대비해, 그 변경된 내용을 되돌린다는 의미가 된다.

            <T1, START>
            <T1, A, 100, 70>
            <T1, COMMIT>

            <T2, START>
            <T2, B, 50, 30>

            장애 발생


|트란잭션|사태|처리
|---|---|---|
|T1|COMMIT 있음|REDO|
|T2|COMMIT 없음|UNDO|

    |저장 프로시져 Stored Procedure|SQL 작업 절차를 저장한 것|
    |사용자 정의 함수 Function|값을 계산해서 반환하는 객체|
    |트리거 Trigger|특정 사건 발생 시 자동 실행되는 객체|

    1. 뷰 View
    - 실제 데이터를 저장하지 않는 가상의 테이블이다.

            학생(학번, 이름, 학과, 전화번호, 주민번호)
    모든 사용자에게 해당 정보 전부를 보여주면 개인정보 보호가 어려워지게 되니 이때 사용된다.

        CREATE VIEW 학생_기본정보 AS
        SELECT 학번, 이름, 학과
        FROM 학생;
        
    이렇게 하면 유저는 학생_기본적보 라는 가상의 테이블을 조회하게 된다.

    이로인한 장점은
    1. 보안성 - 민감한 열을 숨길 수 있다.
    2. 편의성 - 복잡한 질의를 단순하게 사용할 수 있다.
    3. 논리적 독립성 - 원본 테이블 구조 변화릐 영향을 줄일 수 있다.

    하지만 뷰는 실제 테이블이 아닌 가상의 테이블 이므로 모든 뷰에서 삽입/수정/삭제가 자유롭지 못하다.

    2. 저장 프로시져 Stored Procedure
    - 여러 SQL 문장을 하나의 절차로 묶어서 DB에 저장한 것이다.

    쉽게 말하면, 자주 사용하는 SQL 작업 묶음을 DB 안에 저장해두고 호출해서 사용하는 것이라 보면 된다. 예를 들어, 매번 회원 등급을 계산하고 포인트를 수정하는 SQL 을 하나의 절차로 만들어서 사용할 수 있다.

    이의 장점은
    1. 재사용성 - 반복 작업 저장 후 호출
    2. 성능 - DB 내부에서 실행되어 효과적일 수 있다.
    3. 보안 - 직접 테이블 접근하는 대신 프로시저만 허용 가능.
    -



    3. 사용자 정의 함수 Function
    - 입력값을 받아서 결과값을 반환하는 저장 객체다.
    이는 즉, 프로시저와의 차이는 "값을 변환하는가" 가 핵심이 되겠다.

    |구분|프로시져|함수|
    |---|---|---|
    |목적|작업 절차 수행|값 계산 후 반환|
    |변환값|없을 수도 있음|보통 반환값 있음|
    |사용 느낌|명령 실행|계산결과 사용|

    즉 함수는 값을 계산해서 반환하는 저장객체 라는 의미를 지녔다.

    4. 트리거 Trigger
    - 특정 사건이 발생하면 자동으로 실행되는 객체다.
    이는 유니티에서 OnTriggerEnter2D 작업할 때를 생각해보면 이해하기 쉽다.

    트리거는 쉽게 반응하는 사건이 있다.

    |사건|뜻|
    |---|---|
    |INSERT|삽입발생|
    |UPDATE|수정발생
    |DELETE|삭제발생

    이러란 조건들 일때 트리거는 자동으로 실행된다.
    정말로 Unity 게임같다고 느껴지는 부분이 아닐 수 없다.
